Baseline framework

Đề xuất framework mới để đánh giá effort, chất lượng và hiệu quả quy trình phát triển

Framework mới chuẩn hóa toàn bộ chuỗi đánh giá: phân loại ticket, thu thập effort, xây dựng baseline, so sánh lệch chuẩn và RCA cải tiến.

Khung thay đổi cốt lõi

  • Phân loại ticket theo Complexity + Risk / Uncertainty
  • Đo effort theo end-to-end workflow
  • Xây dựng baseline theo CxRy và theo dõi rolling trend
  • So sánh với baseline để phát hiện warning / abnormal
  • Thực hiện RCA để cải thiện process và performance

Phạm vi thay đổi

  • Data: đổi cách thu thập effort
  • Evaluation: đổi cách phân loại ticket
  • Baseline: đổi cách xây dựng mức tham chiếu
  • Comparison: đổi cách xác định normal / warning / abnormal
  • Action: đổi cách review và cải tiến sau đánh giá

02
Pain points

5 điểm khiến baseline cũ khó làm căn cứ đánh giá hiệu quả

1

Chủ quan

CRE phụ thuộc đánh giá cá nhân developer; khi tổng hợp theo % effort có thể không phản ánh đúng phần việc khó.

2

Thiếu validation

Trọng số CRE chưa được kiểm chứng là phản ánh đúng độ khó thực tế.

3

Grouping cứng

S1–S5 chia theo range cố định, dễ dồn ticket vào ít nhóm và làm median kém đại diện.

4

Metric hẹp

Chỉ nhìn Coding + Fix bug, chưa phản ánh effort end-to-end của ticket.

5

Không theo trend

Baseline tĩnh không hỗ trợ theo dõi improvement liên tục theo thời gian.

Vì sao đây là vấn đề

Khi baseline chưa đủ tin cậy, việc so sánh effort giữa các ticket sẽ thiếu công bằng, khó phát hiện bất thường và chưa đủ cơ sở để đánh giá khách quan tác động của process, workflow và Cursor IDE.

Hướng chuyển đổi

Chuyển sang baseline động: phân loại theo Complexity + Risk, đo effort end-to-end và theo dõi theo 4 chiều Productivity, Quality, Stability, Performance.

03
New model

Hướng cải tiến: phân loại ticket theo bản chất công việc để so sánh đúng baseline

Nguyên tắc thiết kế

  • Dùng Complexity và Risk / Uncertainty để phân loại ticket.
  • Không dùng effort làm đầu vào phân loại.
  • Chỉ so sánh ticket với baseline của đúng nhóm CxRy.
  • Cập nhật baseline định kỳ để theo dõi xu hướng.

2 yếu tố đầu vào

Yếu tốMô tảThang điểm
ComplexityĐộ phức tạp kỹ thuật / hệ thống1–5
Risk / UncertaintyMức độ không chắc chắn / phụ thuộc1–5
C + R là input để phân loại; effort là output để so sánh với baseline.

CxRy chỉ có giá trị khi được đánh giá theo bộ tiêu chí chung, giúp phân loại ticket nhất quán hơn, làm baseline đáng tin cậy hơn và hỗ trợ estimate dựa trên dữ liệu lịch sử.

Có guideline chung cho từng mức C1–C5 và R1–R5.
Phân loại ticket → Gán vào CxRy → Lấy baseline cùng nhóm → So sánh actual effort → Review / RCA nếu lệch chuẩn

05
Metrics

Bộ metric mới để xây baseline và đánh giá ticket theo group CxRy

Total effort (end-to-end)

  • Study / Investigation
  • Coding
  • Unit test
  • Code review
  • Fix bug trong ticket
→ Phản ánh đầy đủ hơn effort thực tế của ticket

Core metrics cho mỗi group CxRy

  • Mức tham chiếu
    • P50: effort điển hình
    • P75: ngưỡng cao nhưng còn bình thường
    • P90: ngưỡng cảnh báo
  • Độ tin cậy của baseline
    • Count: số lượng ticket trong group
    • Variance = P75 / P50: độ ổn định
    • Outlier rate: % ticket vượt 1.5× median
Percentile method

P75 và P90 dùng Nearest Rank để ngưỡng là giá trị thực trong dữ liệu.

Quality metrics

Bug density = Bug count / Total effort

Rework ratio = Rework effort / Total effort

Mục tiêu: effort giảm nhưng quality không xấu đi

06
Decision rule

Quy tắc phân loại ticket khi so sánh với baseline

Ý nghĩa vận hành

  • Ticket được so sánh với baseline của đúng group CxRy.
  • P75 là ngưỡng cao nhưng còn bình thường.
  • P90 là ngưỡng bất thường cần phân tích nguyên nhân.
  • P25 giúp phát hiện ticket quá nhanh để kiểm tra lại chất lượng hoặc effort log.

Decision rule

  • Actual < P25 → Thấp bất thường, cần review.
  • P25 ≤ Actual ≤ P75 → Bình thường
  • P75 < Actual ≤ P90 → Cần theo dõi
  • Actual > P90 → Bất thường, cần RCA
Ưu tiên baseline

Count ≥ 10 → dùng BL_Recently

Count < 10 → dùng BL_All-time

BL_All-time = 5–9 → chỉ dùng để warning

BL_All-time < 5 → không dùng để kết luận

Lưu ý

Ticket quá nhanh (< P25) chưa chắc là tốt hơn; cần kiểm tra lại quality hoặc effort log.

Ticket > P90 là tín hiệu để review và RCA, không phải kết luận cuối cùng.

08
RCA

Baseline là công cụ phát hiện bất thường và dẫn đến RCA

Khi nào cần RCA / review sâu hơn?

  • Ticket > P90: effort bất thường
  • Ticket < P25: nhanh bất thường, cần kiểm tra quality hoặc effort log
  • Outlier rate của group tăng theo thời gian: dấu hiệu process mất ổn định

Nhóm nguyên nhân

  • CxRy: phân loại sai nhóm công việc
  • Spec: requirement chưa rõ hoặc thay đổi
  • Dev: vấn đề trong execution
  • System: technical debt / hạn chế hệ thống
  • Data: effort log chưa chính xác
Ticket lệch chuẩn → Review → RCA → Xác định nguyên nhân → Cải thiện process

Cursor không phải là yếu tố duy nhất tạo ra improvement. Điều cần đánh giá là liệu toàn bộ quy trình phát triển có vận hành tốt hơn, ổn định hơn và cho chất lượng tốt hơn hay không.

Takeaway

Kết luận và giá trị tổ chức

Framework mới thay đổi điều gì

  • Phân loại ticket theo CxRy thay cho chấm điểm CRE cảm tính.
  • Đo end-to-end effort thay vì chỉ coding + fix bug
  • So sánh ticket với baseline của đúng nhóm công việc
  • Dùng baseline để review, cảnh báo và RCA

Giá trị ở cấp tổ chức

  • Hình thành chuẩn tham chiếu nội bộ cho từng nhóm công việc
  • Giúp estimate và planning dựa nhiều hơn vào dữ liệu lịch sử
  • Giảm phụ thuộc vào cảm tính cá nhân khi lập kế hoạch và review effort
